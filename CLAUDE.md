# fusion-forge

REST service that builds Python 3.12 virtual environments asynchronously via Kubernetes Jobs (managed by a co-located operator) and stores the resulting tar.gz in the fusion-index artifact registry.

## Stack

- **Go 1.25**, **Gin** (REST API), **pgx/v5** (PostgreSQL), **golang-migrate** (schema migrations)
- **PostgreSQL** via pgx/v5 connection pool with hand-written query functions (`internal/db/db.go`)
- **Kubernetes operator** via `sigs.k8s.io/controller-runtime` v0.19.x тАФ manages `CIBuild` CRs
- **K8s SA token auth** тАФ same `internal/api/middleware/auth.go` pattern as fusion-index
- **License**: GPL-3.0

## Commands

```bash
# Dev build
go build ./...

# Run tests
go test ./... -v

# Regenerate deepcopy functions + CRD YAML after changing api/v1alpha1/ types
make generate
# Then copy updated CRDs to cluster:
kubectl apply -f config/crd/bases/

# Build Docker image (inside minikube)
make docker-build

# Build builder image (inside minikube)
make docker-build-builder

# Full minikube deploy cycle
make minikube-deploy

# Port-forward server (keep running in a separate terminal)
kubectl port-forward -n fusion service/fusion-forge 18080:8080 --address 127.0.0.1
```

## Platform Context

- **Namespace**: `fusion` (minikube)
- **fusion-index**: artifact registry at `http://fusion-index-backend.fusion.svc.cluster.local:8080`; stores venvpacks as artifacts with semver versions
- **Artifact naming**: `venv.{name}` (e.g., `venv.mypackage`) тАФ created in fusion-index on first build
- **Builder image**: `fusion-venv-builder:local` тАФ built from `builder/Dockerfile`; uses `forge-builder` Go binary + Python 3.12
- **Image tags in minikube**: `k8s/deployment.yaml` uses `:local` tags; Makefile default is `:latest` тАФ override with `make docker-build IMG=fusion-forge:local` and `make docker-build-builder BUILDER_IMG=fusion-venv-builder:local`

## CIBuild CRD (`build.fusion-platform.io/v1alpha1`)

```yaml
apiVersion: build.fusion-platform.io/v1alpha1
kind: CIBuild
metadata:
  name: forge-venv-<id>
  namespace: fusion
spec:
  builderImage: fusion-venv-builder:latest
  indexBackendURL: http://fusion-index-backend.fusion.svc.cluster.local:8080
  artifactName: mypackage        # display name
  artifactVersion: 1.0.0         # semver
  description: "..."
  configData:
    requirements.txt: |          # files mounted at /workspace/<filename>
      numpy==1.26.0
  env:
    - name: ARTIFACT_ID          # numeric fusion-index artifact ID
      value: "42"
    - name: ARTIFACT_VERSION
      value: "1.0.0"
    - name: VENV_NAME
      value: "mypackage"
status:
  phase: Pending | Building | Succeeded | Failed
  jobName: forge-job-forge-venv-<id>
  configMapName: forge-cfg-forge-venv-<id>   # cleared after terminal phase
  message: ""
  startedAt: ...
  completedAt: ...
```

## Flow

**POST /api/v1/venvs:**
1. Validate `requirements.txt`
2. `FindOrCreateArtifact("venv.{name}")` тЖТ fusion-index artifact ID
3. Check version not already in fusion-index
4. `CreateVersion(artifactID, semver)`
5. `INSERT venv_build` (PENDING) тЖТ get `id`
6. `UPDATE venv_build SET ci_build_name = forge-venv-{id}`
7. `kubectl create CIBuild forge-venv-{id}` (spec includes ARTIFACT_ID env var)
8. Return `202 Accepted`

**Operator reconcile (CIBuild controller):**
1. On new CR (phase empty): create ConfigMap + Job, set status.phase = Building
2. On Job Succeeded: delete ConfigMap, set phase = Succeeded
3. On Job Failed: delete ConfigMap, set phase = Failed + message
4. No DB access тАФ operator is K8s-native only

**POST /api/v1/gitbuilds:**
1. `FindOrCreateArtifact("venv.{name}")` тЖТ fusion-index artifact ID
2. Check version not already in fusion-index
3. `CreateVersion(artifactID, semver)`
4. `INSERT venv_build` (build_type='git', PENDING) тЖТ get `id`
5. `UPDATE venv_build SET ci_build_name = forge-git-{id}`
6. `kubectl create CIBuild forge-git-{id}` (BuildType=git, GitSource spec, env vars)
7. Builder clones repo, validates structure, builds wheel, installs into venv, uploads archive + optional entrypoint
8. Return `202 Accepted`

**GET /api/v1/venvs/{id} (lazy sync):**
1. Read DB row
2. If status PENDING or BUILDING: read CIBuild CR, map phase тЖТ DB status
3. If changed: `UPDATE venv_build SET status=...`
4. Return response

## Gotchas

- **API uses multipart form, not JSON**: `POST /api/v1/venvs` and `POST /api/v1/venvs/validate` expect `multipart/form-data` with fields `name`, `version`, `description`, and `requirements` (file upload) тАФ not a JSON body
- **Smoke test after deploy**: `curl http://localhost:18080/q/health/live` (live), `curl http://localhost:18080/q/health/ready` (ready), `curl -F "name=t" -F "version=1.0.0" -F "requirements=@/tmp/req.txt" -X POST http://localhost:18080/api/v1/venvs/validate`
- **controller-gen required for type changes**: after editing `api/v1alpha1/`, run `make generate` then `kubectl apply -f config/crd/bases/`; if controller-gen is unavailable, manually update `zz_generated.deepcopy.go` тАФ pointer fields need nil-guard + recursive DeepCopy, value-only structs use `*out = *in`
- **r.Get() zeroes TypeMeta**: set it explicitly before `ctrl.SetControllerReference()` тАФ see `ciBuildWithGVK` pattern in `cibuild_controller.go`
- **client.MergeFrom base**: capture immediately after `r.Get()`, before any mutations; used by `Status().Patch()`
- **DB-only migration**: if upgrading from the Java version, drop the old `venv_build` table and `venv_build_seq` sequence before first run (the new migration uses `BIGSERIAL` not a custom sequence)
- **golang-migrate uses lib/pq**: `DB_SSLMODE=disable` must be set; the DBURL has `?sslmode=disable` appended
- **builder binary is statically linked**: `CGO_ENABLED=0` тАФ the Go binary inside the Python image has no C dependencies
- **Job+ConfigMap ownership**: both are owned by the CIBuild CR via `ctrl.SetControllerReference` тАФ they are GC'd automatically when the CIBuild is deleted; explicit ConfigMap deletion on terminal state is still done to free the requirements.txt content immediately
- **Semver required**: the `version` field in `CreateVenvRequest` must be valid semver (e.g. `1.0.0`) тАФ fusion-index stores versions as `major.minor.patch` integers
- **git build endpoints use JSON, not multipart**: `POST /api/v1/gitbuilds` and `POST /api/v1/gitbuilds/validate` take a JSON body тАФ only `/venvs` endpoints use multipart form
- **Validate endpoints must be read-only**: use `FindArtifact` (GET-only) not `FindOrCreateArtifact` in validate handlers тАФ validate must never create artifacts as a side effect
- **`pip wheel` must not use `--no-deps`**: required so pip can fetch build-system deps (`hatchling`, `flit`, etc.) declared in `pyproject.toml [build-system]`
- **CIBuild CR name prefixes**: `forge-venv-{id}` for requirements builds, `forge-git-{id}` for git builds тАФ must stay distinct to avoid K8s name collisions
- **`UpdateCIBuildName` is a hard failure path**: if the DB update to store the CR name fails, mark the build FAILED immediately and return 500 тАФ do not proceed to create the CIBuild CR, as lazy sync will be permanently broken
- **jobbuilder ConfigData guard**: volumes/mounts are only created when `len(Spec.ConfigData) > 0` тАФ git builds pass empty ConfigData, so this guard prevents mounting a ConfigMap with no keys
- **`metadata_source` three modes**: `"manual"` (default, name+version from request), `"version"` (name from request, version from pyproject.toml), `"full"` (both from pyproject.toml) тАФ server fetches pyproject.toml via go-git in-memory clone before creating the DB row, so the DB record is always consistent
- **`go-git` in-memory clone for pyproject.toml**: `internal/gitutil/pyproject.go` тАФ tries tag ref first, then branch ref; uses `pelletier/go-toml/v2` (already an indirect dep) to parse `[project].name` and `[project].version`; dynamic versions (`dynamic = ["version"]`) are not supported
- **Adding a field to git builds touches 8 layers**: dto/requests.go тЖТ gitbuilds handler (validate + wire) тЖТ db.go (struct + params + scanCols + scan + INSERT) тЖТ internal/gitutil/pyproject.go (if metadata-related) тЖТ api/v1alpha1/cibuild_types.go (GitSourceSpec) тЖТ config/crd/bases/ YAML (gitSource properties, manual edit) тЖТ jobbuilder (env var injection) тЖТ builder/main.go (env var read + use)
- **`strPtr` in handlers/helpers.go already returns nil for empty string** тАФ no need for a separate helper when storing optional string fields as `*string`
- **`project_dir` monorepo support**: optional relative path within a repo; validated to reject absolute paths and `..` escapes; shifts the project root for pyproject.toml lookup (server-side), structure validation, wheel build, and entrypoint resolution (builder-side)

## Validation

`validation.Validate(requirementsTxt, rules)` is a pure function (no side effects). Rules are loaded at startup via `LoadRules(path)` which embeds the default `forge-rules.yaml` via `//go:embed`.

**Always-on rules** (same as Java version):
- Pip options (`-r`, `-e`, `--flag`) rejected
- VCS/URL dependencies rejected
- PEP 508 package name required
- Version specifier required

**Configurable rules** (`forge-rules.yaml`):
```yaml
require-exact-pinning: true   # true = only == accepted
banned-packages: []
max-packages: 100
```
